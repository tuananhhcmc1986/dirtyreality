AI-Native Business Analysis Operating System (BA-OS)

Purpose

This document defines the operating system for Business Analysis used by the BA team.
The system combines:

  • structured requirement engineering
  • AI reasoning
  • semantic knowledge retrieval
  • repository governance
  • human-in-the-loop validation

The objective is to transform requirements from documents into a reasoning system.

Core Objectives

The BA-OS is designed to achieve four strategic outcomes.

1. Speed: Accelerate requirement analysis and specification generation.
2. Quality: Reduce ambiguity, hidden assumptions, and requirement defects.
3. Knowledge Retention: Preserve domain knowledge across projects and time.
4. Reasoning: Enable AI to analyze, compare, and reason over requirements.

Requirement Structure Standard

Each requirement is stored as an atomic node.

REQ-142
Version: 1.1
Type: Business Rule

Context
When creating shipment orders.

Rule
Shipment must not be created if available_stock < reserved_stock.

Entities
Shipment, Inventory

Data Fields
available_stock, reserved_stock

Dependencies
Inventory Service

Related Artifacts
UIF-033, DATA-INV-02

Requirement Versioning

Versioning must occur at the requirement level, not only document level.
Example: REQ-142 v1.0, REQ-142 v1.1, REQ-142 v2.0
Benefits: traceability, change reasoning, historical knowledge.

Stage 1 — Stakeholder Elicitation

Objective: Capture domain knowledge from stakeholders.
Input: meetings, workshops, process walkthroughs.
Output: Meeting transcripts or structured notes.

Stage 2 — AI Requirement Extraction

Prompt Template
Act as a senior business analyst. Analyze the following stakeholder conversation.
Tasks:

1. Extract explicit requirements
2. Identify implicit assumptions
3. Detect ambiguities
4. Generate clarification questions
5. Detect possible requirement conflicts

Output: Extracted Requirements, Assumptions, Ambiguities, Clarification Questions, Potential Conflicts.
Human BA validates the result.

Stage 3 — Context Retrieval (RAG Layer)

Before analyzing new requirements, retrieve related knowledge.
Pipeline: `Draft Requirement` → `Vector Search` → `Retrieve Related Requirements` → `Attach Context to Prompt`.

Stage 4 — Requirement Structuring

Convert extracted requirements into structured format. Key components: context, rule, entities, data fields, dependencies. Structured requirements enable AI reasoning.

Stage 5 — Specification Drafting

Requirements are expanded into module specifications. Standard sections: Functional Requirements, Process Flow, Business Rules, Data Requirements, Integration Points, Exception Handling.

Stage 6 — Requirement Stress Testing

Before committing specifications, run AI stress tests. Prompt: Act as a system architect. Stress test the following requirements. Identify: ambiguities, missing edge cases, logical contradictions, data integrity risks, operational failure scenarios.

Stage 7 — Conflict Detection

Conflict detection occurs in three layers.

1. Layer 1 — Semantic Similarity: Vector search finds similar requirements.
2. Layer 2 — Structural Filtering: Filter by entity, module, data fields.
3. Layer 3 — AI Reasoning: AI evaluates logical contradictions.

Stage 8 — Dependency Detection

Dependencies are identified between requirements, services, data entities, UI flows.
Example: REQ-142 `depends_on` Inventory Service.

Stage 9 — Repository Governance

Process: `BA review` → `AI validation` → `Lead BA approval` → `Repository commit`. Version history is automatically recorded.

Stage 10 — Embedding Pipeline

Each atomic requirement becomes an embedding. 1 requirement = 1 embedding.
Metadata example: requirement_id, version, module, entity, data_fields, status.

Stage 11 — Vector Knowledge Layer

Vector database enables semantic retrieval. Capabilities: similar requirement search, cross-spec comparison, knowledge reuse, dependency discovery.

Stage 12 — Impact Analysis

Requirement changes trigger impact detection. Pipeline: `Requirement change` → `Vector similarity search` → `Related requirement detection` → `AI reasoning` → `Impact report`.

Stage 13 — Decision Log

Architectural decisions are stored explicitly.
DEC-014
Context: Inventory validation conflict discovered.
Decision: Shipment must validate reserved stock.
Rationale: Prevent overselling.
Decision logs preserve reasoning history.

Stage 14 — Knowledge Graph Layer (Future)

Requirements can form a graph structure.
Nodes: Requirement, Entity, Process, UI Flow, System.
Edges: depends_on, produces, consumes, conflicts_with.
Graph + vector search enables advanced reasoning.

BA Role Model

  • BA: stakeholder elicitation, requirement structuring, AI prompt orchestration, specification drafting.
  • Senior BA: requirement quality control, conflict detection, dependency analysis, domain rule governance.
  • Lead BA: repository governance, architecture decisions, AI workflow improvement, knowledge system maintenance.

AI Layer Responsibilities

  • AI performs: requirement extraction, ambiguity detection, clarification generation, conflict detection, stress testing, impact analysis.
  • Human BA performs: judgment, domain interpretation, decision making, governance.

Spec Repository CI Pipeline

The spec repository behaves similarly to a code repository.
Pipeline: `Spec Commit` → `AI Requirement Stress Test` → `Conflict Scan` → `Dependency Detection` → `Impact Analysis` → `Approval`.

Prompt Library

  • Extraction Prompt: Extract requirements from conversation. Identify ambiguity and assumptions.
  • Stress Test Prompt: Challenge requirement logic. Identify edge cases and operational failures.
  • Conflict Detection Prompt: Compare two requirements. Determine whether they conflict logically.
  • Impact Analysis Prompt: Analyze how requirement changes affect related modules.

Long-Term Evolution

As the repository grows, the system becomes a Requirement Intelligence Platform. Capabilities evolve toward automated requirement reasoning, organizational domain memory, predictive dependency detection, AI-assisted architecture governance.

Final Principle

Requirements are not documents.
Requirements are structured knowledge.

Khi được cấu trúc đúng cách, chúng trở thành một hệ thống tri thức mà AI có thể reasoning trên đó.
